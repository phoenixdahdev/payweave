- bb568a6: `BillingSync` — `payweave.sync`.

- `payweave.sync` pushes every configured plan's definition to the database and each configured billing-capable provider: Stripe gets a Product + recurring Price per paid plan; Paystack gets a Plan (`POST /plan`); free/default plans are DB-only (zero provider objects, zero provider calls). Provider refs land back in `pw_plans.provider_refs` per version, ready for `subscribe` to read.
- Idempotent by content hash, two layers deep: an unchanged plan makes ZERO provider API calls on a re-push (asserted via MSW call-counting); a hash miss searches for a `pwv_plan`/`pwv_hash`-tagged object BEFORE ever creating one, adopting it instead of creating a duplicate — the crash-between-provider-create-and-`pushVersion` recovery path. Stripe's Product (`name`) and Price (`amount`/`currency`/`interval`) are diffed independently, so a display-name-only edit refreshes the Product in place with zero Price churn; a price change creates a NEW Price and archives the old one (`active: false`) — never deletes. Paystack has no Product/Price split and no update endpoint on the in-scope resource surface, so any content change (including a name-only edit) creates a new Plan, tagged for adoption via its `description` field (Paystack's Plan API has no `metadata` field to use instead).
- Runtime guard: `sync` throws `PayweaveConfigError` without both `database` and `products` configured.
- `pushPlanToProvider` (exported from `src/products/sync.ts`) is the per-provider dispatch `sync`'s loop drives — calling it directly for a non-billing-capable provider (Flutterwave) throws a typed, documented `PayweaveConfigError` rather than guessing a mapping; `sync` itself silently skips a configured Flutterwave key and reports it back via `SyncResult.skippedProviders`.
- ⚠️ resolutions recorded in `plans-and-features.md` (verified 2026-07-13): Paystack's real Plan API supports in-place amount updates via `PUT /plan/{id}`, but the in-scope `src/paystack/resources/plans.ts` surface has no `update` method — a follow-up ticket adding one would let this engine do true in-place price rotation instead of creating a new plan every time. Flutterwave's v3 Payment Plans API (`POST /v3/payment-plans`) is now confidently mapped (amount is confirmed immutable after creation — a price change needs a new plan, same shape as Stripe/Paystack), but is deferred rather than shipped: neither its create nor update endpoint documents a metadata/tagging field for crash-resume adoption, and no Flutterwave payment-plans resource/schema exists in this codebase yet (a new Surface A module, out of this ticket's scope) — consistent with `subscribe.ts`'s existing `BILLING_CAPABLE_PROVIDERS` exclusion.
- Mirrors `src/products/subscribe.ts`'s shape: reuses its `BillingContext`, `BILLING_CAPABLE_PROVIDERS`, and `resolvePlanGroup` rather than duplicating them.

- bb568a6: `check` + `report` metered usage.

- `payweave.check({ customerId, featureId, consume? })` and `payweave.report({ customerId, featureId, amount? })` are real now — the billing module lives in `src/products/usage.ts`, mirroring `subscribe.ts`'s `BillingContext` seam. Resolution order: the customer's active subscription for the feature's group → that plan's inclusion; no subscription → the group's default plan's inclusion (no rows required); feature not included in the resolved plan → `allowed: false` / `balance`/`limit`/`resetsAt: null` — a normal "upsell" answer, never an error.
- Both bottom out in exactly ONE `database.balances.consume` call — the SDK layer adds zero atomicity of its own. `check` is `consume(amount: 0)` (a peek that still performs the lazy reset — row creation + reset happen even at `amount: 0`); `report` is `consume(amount, conditional: false)` (unconditional — never throws for "over limit"); `check({ consume: true })` is `consume(amount: 1, conditional: true)` — the documented atomic gate: allowed attempts decrement, denied attempts leave the row untouched, in ONE operation (never a separate check-then-consume pair).
- Lazy creation + lazy reset: a never-touched (customer, feature) balance is created on first `check`/`report`, seeded from the resolved plan's `limit`/`reset`; a paid plan's balance anchors at the subscription's `current_period_start`, while a default/free plan's balance anchors at the customer's own first use. When the current period has rolled over (via `src/products/period.ts`'s `currentPeriod`), the adapter resets the balance to the fresh limit before applying the operation — no cron jobs, no SDK-side read-modify-write.
- `CheckResult` is now `{ allowed, balance, limit, resetsAt, planId }` and `ReportResult` is `{ balance, resetsAt }` (this replaces the earlier placeholder `{ allowed, remaining? }` / `{ remaining }` shapes — pre-1.0, this is not previously-published-behavior a downstream caller could depend on, since `check`/`report` never resolved before this change). `ReportInput.amount` is now optional, defaulting to `1`.
- Runtime guards: `PayweaveConfigError` when `database` isn't configured; `PayweaveValidationError` for an unknown `featureId`, `report` on a boolean feature, `report` on a feature the resolved plan excludes, a non-positive/non-integer `amount`, or an active subscription referencing a plan no longer in `products`; `PayweaveConfigError` for a group with no default plan and no active subscription (a configuration gap, not a normal "no access" answer).
- Spec-silent decisions recorded in `src/products/usage.ts`'s module doc comment: limits/reset intervals are resolved from the in-memory `ctx.products` (keyed by the subscription's `planId`), not a `database.plans.getActiveVersion` round trip, since the adapter contract only exposes the latest pushed version per plan id either way; a feature id appearing under more than one group resolves to the first occurrence in `products` array order; the anchor-seeding split is on the resolved plan's price (paid vs free), not on subscription-existence.

- bb568a6: `payweave listen`: a local webhook relay for development. Starts a plain `node:http` server (no bundled dependency at all) that receives provider webhooks on `--port` (default `4242`), verifies each one through the SAME `payweave.webhooks.constructEvent` dispatch the SDK's own webhook handlers use (raw bytes, never re-parsed before verification), prints the normalized event, and either forwards the EXACT raw bytes + pass-through headers to your app (`--forward-to <url>` — never re-serialized, so the dev's own signature verification also passes) or applies the event directly (`event.apply`). `--provider <id>` scopes processing to one configured provider (an unconfigured id is a usage error; other providers' webhooks are still acknowledged, just skipped). `--retry <window>` (`5m`/`30s`/`none`) redelivers this session's own failed deliveries for up to that long after receipt — the local-relay design (`cli.md` resolved ⚠️ tunnel decision) has no provider-side event log to backfill from, so retry is necessarily in-process and at-least-once; without it, a failed delivery is logged once and dropped. `listen` refuses to start against a config whose environment resolves to `live` unless `--live` is passed, in which case it proceeds with a loud warning (dev-only tool). `-- <cmd>` spawns your dev server alongside, inheriting stdio and propagating its exit code; SIGINT (or the child exiting) triggers a graceful shutdown. This replaces the placeholder that unconditionally exited 1 naming this ticket.

- 6a166ab: Initial Payweave SDK — a unified, fully-typed TypeScript SDK for Paystack and Flutterwave.

- **Two surfaces:** provider-native Surface A (every P0 endpoint, 1:1 with each provider's fields, compile-time narrowed on `provider`) and a normalized Surface B unified layer (`checkout`, `verify`, `refunds`, `transfers`, `banks`) with always-minor-unit Money and a `raw` escape hatch.

- bb568a6: `feature` + `plan` primitives on the `payweave/products` subpath.

- `feature({ id, type: "boolean" | "metered" })` validates the id (`^[a-z0-9][a-z0-9_-]{0,63}$`) and returns a callable: a boolean feature is called with no arguments (`proModels`), a metered feature requires `{ limit, reset }` (`messages({ limit: 100, reset: "month" })`). - bb568a6: `event.apply` — webhook → billing state.

- Every `WebhookEvent` returned by `constructEvent`/`payweave.webhooks.constructEvent` now carries `apply`: idempotently applies the event's billing-state transition — `subscription.created`/`subscription.updated` → status + real period bounds (Stripe reads the period from subscription ITEMS on the pinned API version, not the subscription object itself; Paystack from `next_payment_date`); `payment.succeeded`/`invoice.paid` → flips the matching `incomplete` row (created by `subscribe`) to `active`, or refreshes an already-active row's period on renewal; `subscription.canceled` → marks the row `canceled` (the customer falls back to the group default by absence); `invoice.payment_failed` → `past_due`. Unmapped/irrelevant events (`"unknown"`, transfers, refunds, disputes, …) are a safe no-op — never thrown.
- Idempotency rides `webhookEvents.claim` alone: a replayed event applies exactly once, whether replayed as the same `WebhookEvent` object or reconstructed fresh from an identical redelivered payload. Claim + the state mutation + `markApplied` run inside ONE `database.transaction(...)` so a failed apply rolls the claim back; a claimed-but-never-applied event (crash between claim and `markApplied`) becomes re-claimable after `staleClaimAfterMs`, never before.
- Out-of-order guard: an update event whose period is OLDER than what's already stored is rejected as stale and never applied; a late `subscription.updated` arriving after `subscription.canceled` finds no row in the active set and safely no-ops rather than resurrecting it.
- `apply` always exists on the returned event (mirrors `subscribe`/`check`/`report`) but throws `PayweaveConfigError` when called without a `database` configured.
- Plan-change handling: when an update event's plan differs from the row's stored plan, `apply` patches `planId`/`planVersion` and resets every metered feature's balance to the new plan's limits (`balances.resetTo`) — `resetBalancesForPlanChange` (`src/products/apply.ts`) is exported standalone so the conformance scenario can wire/call it directly too.
