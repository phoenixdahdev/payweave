`check()` + `report()` metered usage (PW-902, metered-usage.md §1–§7).

- `payweave.check({ customerId, featureId, consume? })` and `payweave.report({ customerId, featureId, amount? })` are real now — the billing module lives in `src/products/usage.ts`, mirroring `subscribe.ts`'s `BillingContext` seam. Resolution order (§4): the customer's active subscription for the feature's group → that plan's inclusion; no subscription → the group's default plan's inclusion (no rows required); feature not included in the resolved plan → `allowed: false` / `balance`/`limit`/`resetsAt: null` — a normal "upsell" answer, never an error.
- Both bottom out in exactly ONE `database.balances.consume` call — the SDK layer adds zero atomicity of its own. `check` is `consume(amount: 0)` (a peek that still performs the lazy reset — row creation + reset happen even at `amount: 0`); `report` is `consume(amount, conditional: false)` (unconditional — never throws for "over limit"); `check({ consume: true })` is `consume(amount: 1, conditional: true)` — the documented atomic gate (§6): allowed attempts decrement, denied attempts leave the row untouched, in ONE operation (never a separate check-then-consume pair).
- Lazy creation + lazy reset (§5): a never-touched (customer, feature) balance is created on first `check`/`report`, seeded from the resolved plan's `limit`/`reset`; a paid plan's balance anchors at the subscription's `current_period_start`, while a default/free plan's balance anchors at the customer's own first use. When the current period has rolled over (via PW-901's `currentPeriod`), the adapter resets the balance to the fresh limit before applying the operation — no cron jobs, no SDK-side read-modify-write.
- `CheckResult` is now `{ allowed, balance, limit, resetsAt, planId }` and `ReportResult` is `{ balance, resetsAt }`, matching metered-usage.md §3 verbatim (this replaces PW-802's placeholder `{ allowed, remaining? }` / `{ remaining }` shapes — pre-1.0, this is not previously-published-behavior a downstream caller could depend on, since `check`/`report` never resolved before this change). `ReportInput.amount` is now optional, defaulting to `1`.
- Runtime guards: `PayweaveConfigError` when `database` isn't configured (unified-config.md §3); `PayweaveValidationError` for an unknown `featureId`, `report()` on a boolean feature, `report()` on a feature the resolved plan excludes, a non-positive/non-integer `amount`, or an active subscription referencing a plan no longer in `products`; `PayweaveConfigError` for a group with no default plan and no active subscription (a configuration gap, not a normal "no access" answer).
- Spec-silent decisions recorded in `src/products/usage.ts`'s module doc comment: limits/reset intervals are resolved from the in-memory `ctx.products` (keyed by the subscription's `planId`), not a `database.plans.getActiveVersion` round trip, since the adapter contract only exposes the latest pushed version per plan id either way; a feature id appearing under more than one group resolves to the first occurrence in `products` array order; the anchor-seeding split is on the resolved plan's price (paid vs free), not on subscription-existence.
