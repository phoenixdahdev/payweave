Land the Drizzle ORM `DatabaseAdapter` (database.md §4, PW-708) at `payweave/db/drizzle`. `drizzleAdapter(db, options?)` wraps your existing `drizzle-orm` instance — postgres, mysql, or sqlite/libSQL — auto-detecting the dialect via `instanceof` against `drizzle-orm`'s own dialect base classes (`options.dialect` is an explicit override). Published schema files ship for all three dialects (`pgSchema`/`mysqlSchema`/`sqliteSchema`, re-exported from `payweave/db/drizzle`) — merge one into your own Drizzle schema and migrate with `drizzle-kit` yourself: "same model" as the Prisma adapter, `migrations.apply()` never shells out or runs DDL, returning an instructions result instead; `migrations.status()` reports missing tables by probing the connected database. `balances.consume`/`webhookEvents.claim`/`plans.pushVersion` are atomic (postgres/mysql via `db.transaction()` + `SELECT ... FOR UPDATE`; sqlite via a single-connection FIFO queue, since neither `better-sqlite3` nor `@libsql/client` supports concurrent async `drizzle-orm` transactions safely) — verified in-process against `drizzle-orm/libsql` over an in-memory `@libsql/client` (PW-702's full conformance suite, including the §5 N=50 concurrency race); the postgres/mysql code paths are implemented against the same contract but their conformance runs ride PW-710's dockerized CI matrix. `drizzle-orm` is an optional peerDependency (+ devDependency, alongside `drizzle-kit` for tests) — `payweave`'s runtime `dependencies` stay zod-only. `payweave/db/drizzle` no longer throws the PW-505 placeholder error.
