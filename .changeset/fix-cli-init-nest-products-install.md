"payweave": minor
---

Fix several `payweave init` issues, mostly around NestJS:

- **NestJS was never detected** — `detectFramework` only checked for Next.js/Express/Fastify, so a Nest project silently fell through to the generic `node:http` scaffold. Detection now checks `@nestjs/core` before Express, since Nest's default HTTP adapter *is* Express (`@nestjs/platform-express`) and a Nest project very commonly also depends on `express` directly.
- **NestJS now scaffolds a proper feature module, not loose root files** — `payweave.ts` / `lib/payweave-client.ts` aren't how Nest projects are structured. NestJS gets a self-contained `src/payweave/` module instead: `payweave.service.ts` (an `@Injectable` wrapping `createPayweave(...)`), `payweave-webhook.controller.ts` (DI-wired to that service instead of constructing its own client), `payweave.module.ts` tying them together, and a `README.md` documenting the folder. `products.ts` (when a database is configured) lives in the same folder rather than at the project root.
- **`PayweaveModule` is now wired into the project's own `AppModule` automatically** — after scaffolding, `init` locates `app.module.ts` (the `nest new` convention, with a bounded recursive fallback for other layouts) and adds `PayweaveModule` to its `imports` array, tracking bracket depth so a nested call already in that array — e.g. `ConfigModule.forRoot({ load: [...] })` — doesn't get mistaken for the array's own close. Finding no `app.module.ts` is a warning, not a failure; the generated module is still correct and wired in by hand with one line.
- **`products.ts` was scaffolded even with no database selected** — plans/features/metered usage all require a database adapter, so a payments-only project (`database: "none"`) got a `products.ts` it had no way to use. It's now only generated when a real database is configured.
- **The dependency was never actually installed** — the wizard wrote config importing from `"payweave"` but never ensured the target project actually had it as a dependency (`npx payweave init` only downloads the CLI temporarily to run the wizard). It now detects the project's package manager from its lockfile (pnpm/yarn/bun/npm, defaulting to npm) and runs the matching install command after scaffolding. `--no-install` skips this; a failed install warns with the manual command rather than failing the whole run, since the scaffold itself is still correct.
- **`.env.example` was wholesale-overwritten on a second `init` run** — teams fill in real local values there, so re-running `init` (e.g. after adding a provider) would silently discard that work. It's now merged instead: only the blocks whose vars aren't already present get appended, and it's never clobbered — not even with `--force`. Re-running with unchanged answers is a true no-op.
