Land the PostgreSQL `DatabaseAdapter` (database.md §4, PW-704) at `payweave/db/postgres`. `postgresAdapter({ connectionString } | pool)` wraps `pg`, validating a `postgres://`/`postgresql://` connection string eagerly and connecting lazily (an existing `pg` `Pool` instance is also accepted and never closed by the adapter). `balances.consume` and `webhookEvents.claim` are each ONE `INSERT ... ON CONFLICT DO UPDATE ... RETURNING` statement — the lazy-reset, conditional-gate, and decrement arithmetic run inside pg's own row locking within a single round trip, never a read-then-write from the application (database.md §5); `plans.pushVersion`/`customers.linkProviderRef` use a short row-locked transaction, matching the other SQL adapters. Payweave-owned SQL migrations (PW-703's engine) run through a `$n`-placeholder executor. `pg` is an optional peerDependency (+ devDependency, alongside `@types/pg`) — `payweave`'s runtime `dependencies` stay zod-only. `payweave/db/postgres` no longer throws the PW-505 placeholder error. Docker-backed conformance (dockerized `postgres:16`) is registered behind `PW_DB_CONFORMANCE_DOCKER=1` for PW-710's CI matrix; this environment has no docker, so it is proven in-process instead: single-statement shape assertions against a mock `pg` Pool, and a property-tested JS mirror of the reset-due period-math SQL against `src/products/period.ts`'s oracle.
