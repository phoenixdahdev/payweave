`payweave listen` (PW-1006): a local webhook relay for development. Starts a plain `node:http` server (no bundled dependency at all) that receives provider webhooks on `--port` (default `4242`), verifies each one through the SAME `payweave.webhooks.constructEvent` dispatch the SDK's own webhook handlers use (raw bytes, never re-parsed before verification), prints the normalized event, and either forwards the EXACT raw bytes + pass-through headers to your app (`--forward-to <url>` — never re-serialized, so the dev's own signature verification also passes) or applies the event directly (`event.apply()`). `--provider <id>` scopes processing to one configured provider (an unconfigured id is a usage error; other providers' webhooks are still acknowledged, just skipped). `--retry <window>` (`5m`/`30s`/`none`) redelivers this session's own failed deliveries for up to that long after receipt — the local-relay design (`cli.md` §8's resolved ⚠️ tunnel decision) has no provider-side event log to backfill from, so retry is necessarily in-process and at-least-once; without it, a failed delivery is logged once and dropped. `listen` refuses to start against a config whose environment resolves to `live` unless `--live` is passed, in which case it proceeds with a loud warning (dev-only tool). `-- <cmd>` spawns your dev server alongside, inheriting stdio and propagating its exit code; SIGINT (or the child exiting) triggers a graceful shutdown. This replaces the PW-1001 placeholder that unconditionally exited 1 naming this ticket.
