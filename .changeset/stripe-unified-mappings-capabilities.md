Stripe joins the unified-layer status/event mapping tables in `unified/mappings.ts` (PW-607, providers.md §3.3/§3.5), and a unified-ops capability matrix lands alongside them.

- `MappingProvider` widens (additively) to include `"stripe"`. `STRIPE_STATUS_MAP` normalizes Checkout Session `payment_status`/`status` and PaymentIntent `status` to `UnifiedStatus`; `STRIPE_EVENT_MAP` + `STRIPE_EVENT_STATUS_SPLIT_MAP` normalize Stripe webhook event names to `UnifiedEventType`, including the delayed/async-payment nuance verified against the official docs (`checkout.session.completed` only counts as `payment.succeeded` when `payment_status` is `paid`/`no_payment_required`; the async success/failure signal for delayed payment methods arrives via `checkout.session.async_payment_succeeded`/`.async_payment_failed`, both newly mapped).
- `UnifiedEventType` gains four new members — `subscription.updated`, `subscription.canceled`, `invoice.paid`, `invoice.payment_failed` — added for ALL THREE providers in this PR so the unified vocabulary stays total: new Paystack rows (`subscription.disable`, `subscription.not_renew`, `invoice.update`, `invoice.payment_failed`) and new Flutterwave rows (`refund.completed`, `subscription.cancelled`, `chargeback.initiated`, resolving the standing `NOTE(verify)`), each verified against the provider's official docs.
- New unified-ops capability matrix: `UNIFIED_CAPABILITY_MATRIX`, `UnifiedOpName`, `UnifiedOpCapability`, `isUnifiedOpSupported`, and `assertUnifiedCapability` (providers.md §3.3) — data describing which of the six unified ops (`checkout.create`, `verify`, `refunds.create`, `transfers.create`, `banks.list`, `banks.resolveAccount`) each provider supports. Paystack and Flutterwave support all six; Stripe does not support `transfers.create`/`banks.*` in v1 (Stripe payouts/Connect transfers are semantically different from Paystack/FLW bank transfers, and bank lookup is NG-specific).
- `payweave.capabilities()` on the `createPayweave` client root: called with no argument, returns the capability matrix for every CONFIGURED provider; called with a configured provider name, returns just that provider's per-op matrix; called with an unconfigured provider name, throws `PayweaveConfigError`.
- Calling `transfers.create`/`banks.list`/`banks.resolveAccount` routed to stripe (via `defaultProvider` or a per-call `provider` override) now throws the typed `PayweaveValidationError` naming the provider + operation (e.g. `"transfers are not supported on stripe"`), synchronously, before any request would be sent — this REPLACES the previous "PW-607 pending" rejection PW-502 added for those two ops. `checkout.create`/`verify`/`refunds.create` on stripe remain capability-supported per providers.md §3.3, but the unified implementation that wires them onto the real Stripe resources (`src/unified/stripe.ts`) has not shipped yet — calling them still rejects, now with an honest "not implemented yet" message distinct from a capability gap, rather than a "PW-607" placeholder.

All additions extend the public `unified/mappings.ts` contract (AGENTS.md §9) — no existing native→unified row, status value, or capability cell changed meaning.
