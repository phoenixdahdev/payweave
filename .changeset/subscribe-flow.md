`subscribe()` billing flow (PW-804, plans-and-features.md §11).

- `payweave.subscribe({ customerId, planId, provider?, successUrl?, cancelUrl? })` is real now — the billing module lives in `src/products/subscribe.ts` and implements every §11 branch: customer upsert + provider-ref linking (create the provider customer once per provider, `linkProviderRef` persists it); paid plans create a provider checkout (Stripe Checkout Session in `subscription` mode against the pushed price, Paystack `transactions.initialize` against the pushed plan code) and a local `incomplete` row — never locally activated here, that's PW-805's `event.apply()`; free non-default plans activate a local `active` row immediately with zero provider calls; a group's default plan is a successful no-op (`{ status: "active", planId, subscription: null }`) absent an active sub in the group, and throws the spec's exact group-exclusivity error (`customer already has an active plan in group '<group>'`) otherwise.
- Runtime guard: `subscribe()` throws `PayweaveConfigError` when `database` isn't configured (unified-config.md §3), and when the resolved/overridden `provider` isn't configured or isn't billing-capable (v1: `stripe`/`paystack` only — Flutterwave's payment-plan mapping is ⚠️ pending PW-803).
- `PayweaveClient` now exposes `database`/`products` (the resolved adapter and minor-units-resolved products) as read-only root properties — additive PW-802 follow-up; `src/cli/status.ts`'s duck-typed workaround picks these up automatically.
- `check`/`report` are unchanged — still `PayweaveConfigError` stubs, landing with PW-902.
- Seams left for sibling tickets: `resolvePlanGroup`, `BILLING_CAPABLE_PROVIDERS`, and the `BillingContext` shape are exported from `src/products/subscribe.ts` for PW-803 (billing sync), PW-805 (webhook → billing state), and PW-902 (`check`/`report`) to reuse; the module's doc comment records the exact correlation data left for PW-805 (Stripe `client_reference_id`/`metadata.pwv_reference`, Paystack's caller-supplied `reference`) and the spec-silent decisions made along the way (groupless-plan → singleton group; a second `subscribe()` on a pending checkout creates a new `incomplete` row rather than superseding, since the DB contract has no read method for pending rows; Paystack's required `email` is read off the local `pw_customers` row since `subscribe()`'s input carries none).
